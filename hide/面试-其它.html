AIDL理解

Binder

当我们想要绑定一个 Service 的时候，该 Service 必须是要在 ServiceManager 里面注册过了的，我们调用了 bindService() 之后，ServiceManager 就会寻找目标 Service，然后向 Client 端返回一个 BinderProxy 对象，这个 对象通过Binder驱动来代理一切 Client 和 Service 的通信工作，Binder驱动会自动完成BinderProxy和Binder之间的转换。还有一个需要注意的就是，如果是在同一个进程中使用 AIDL 那么，ServiceManager 返回的就不是 BinderProxy 代理对象，而是一个Binder对象，这里是策略模式。

Handler 理解

Handler 机制是用来处理不同线程之前的通信的，也就是异步消息处理机制，具体的实现用到了生产者消费者模式。

当我们在 A 线程，实例化出一个 Handler 对象的时候，源码内部其实做了更多的处理，拿到和当前线程绑定的 Looper 对象，如果没有 Looper 对象，就必须要调用 Looper.prepare() 方法为该线程创建一个，否则会跑出异常， Looper 对象又会创建一个 MessageQueue，MessageQueue 用来存放消息。

如果想要启动消息循环的话，需要调用 Looper 对象的 loop 方法，这样 Looper 就开启了一个无线循环，不断的调用 MessageQueue 的 next 方法从 MessageQueue 中去取 Message，这个 next 方法是一个阻塞式的，如果取不到消息就会一直等待。

当我们在 B 线程，使用 A 线程创建出来的 Handler，发送一个消息的时候，实际上是发送到了线程 A 的 MessageQueue 中，之所以能做到，是因为 Handler 里面保存了 Looper 的引用，Looper 保存了 MessageQueue 的引用。

这样消息就成功的放入到了 MessageQueue 中，那么放入消息队列中的消息，怎么得到处理呢？根据生产者消费者模式，我们知道，肯定要有消费者来队列中取出消息。为了能够取出消息，Looper 提供了一个循环，Looper.loop() 方法，不断的取出消息队列的消息，所以可以认为 Looper 所在的线程 A 就是消费者，Looper 引用的 MessageQueue 是仓库，使用 Handler 的线程 B 是生产者。

那么，取出了消息之后，由于 Message 里面有 Handler 的引用，所以就直接触发了 Handler 的 dispatchMessage() 方法，又调用到 handleMessage() 方法，这就是Handler 机制的整个过程，它利用队列管理消息，消息的处理是有先后的，还支持消息延时。

说的直白一点，就是线程之间要通信，每个线程都维护着自己的一个仓库，别的线程都可以通过某种方法来往仓库里面放东西，而线程自己会不断的去处理仓库里面的东西。只要别的线程将需要通信的东西封装好，然后发往另一个线程的仓库，那么通信的内容就会被那个线程处理，这就是异步消息的模型。

那么还有一个问题，我们平常在使用的时候，没有调用 Looper.prepare() 和 Looper.loop() ，为什么能正常工作？是因为 ActivityThead 替我们做了。

可以看到，主线程其实已经调用了这些方法，想的更深入一些，其实我们的应用从启动开始就是处于一个消息循环之中，界面的绘制等等，都是基于这个的。做过游戏开发的应该更有体验些，整个游戏程序就是处于一个无线循环之中，去不断的绘制各种界面等等。

设计一套图片异步加载缓存方案

因为涉及到内存与磁盘，所以先解释一下 Android 中的几个概念：ROM，RAM，SD卡，虽然现在基本上手机都没有 SD 卡了。

RAM

白话来讲，就是我们常说的真正意义上的内存，就相当于你电脑的内存条。

ROM

简单的说，就是相当于你windows电脑的C盘。每次安装一个程序或者游戏，即便你安装到了SD卡中，但你的ROM空间依然还是会被占用一部分。

SD卡

就是手机的外部存储空间，这个我们可以理解成电脑的D盘、E盘，或者外接移动硬盘也行。

Bitmap 的内存优化

应用的大多数 OOM 都是由 Bitmap 引起的。

例如：使用Pixel手机拍摄4048x3036像素（1200W）的照片，如果按ARGB_8888来显示的话，需要48MB的内存空间（4048*3036*4 bytes），这么大的内存消耗极易引发OOM。

那么如何减少内存的消耗呢？

首先，看看图片在内存中占用的大小是如何计算的？

  1. 使用 BitmapFactory.decodeResource() 这个方法来解码图片，

  2. 那么它占用的内存 = width * height * （nTargetDensity/inDensity）^2 * 一个像素所占的内存 。

  3. 使用 BitmapFactory.decodeFile() 等其它方法，此时Bitmap默认占用的内存 = width * height * 一个像素所占的内存。

所以，如果图片放在 drawable-xxxx 文件夹里面是会进行缩放处理的，所占的内存也不一样。

对图片进行尺寸缩放，使用 inSimpleSize 字段，设置该字段之后，图片读取时，不会直接加载到内存，只会获取该图片的宽和高，由此可以设置缩放比例，相当于对图片进行采样，得到最后按比例缩放之后图片，即内存变成了原来的 1 / 缩放比例的平方 。然后，就是对图片的像素进行处理，显示效果要求不高的话，可以采用 RGB_565 格式，相比 ARGB_8888 内存可以再节约一半。

还有一个方法，就是直接进行图片复用，意思是不同的图片利用同一块内存，但是这里用很多限制，还有一些缺点，就是如果复用的图片显示到 ImageView 的时候，设置了 ScaleType，会相互影响，所以还是采用 Fesco 这些图片加载框架比较好。

关于图片另一种意义上的压缩：Bitmap.compress()

它是在保持像素的前提下改变图片的位深及透明度等，来达到压缩图片的目的，不会减少图片的像素。进过它压缩的图片文件大小会变小，但是解码成bitmap后占得内存是不变的。

Android内存泄露及管理

先看看 JVM 的内存模型图：

关于每个区的具体说明，有链接：http://gityuan.com/2016/01/09/java-memory/

我们通常所说的内存泄露，就是发生在 Heap 中。我们都知道，java 的内存是由 GC 来管理的，基本不需要程序员来插手，先来了解 GC 是如何来管理内存的，再来说说为什么会发生内存泄露。具体的内容看：http://www.jianshu.com/p/c9ac99b87d56

简单的说，判断一个对象能不能回收，就是从 GC ROOT 开始根据引用关系来遍历整个 Heap 区，然后标记，然后用 GC 来回收没有标记的对象。所以说，如果发生了内存泄露，肯定是某个对象该被会回收的时候，发现还有另外对象还持有它的引用，这才导致了无法被 GC 回收。如果内存泄露发生的次数比较多，还会产生 OOM。

常见的内存泄露问题：

  1. Handler 引起的

    Android 应用启动的时候会创建 UI 主线程的 Looper 对象，它存在于整个应用的生命周期，用于处理消息队列里的 Message。而这些 Message 会引用发送该消息的 Handler 对象。

    那么问题来了，如果这些 Handler 是 Activity 的内部类，那么当这些 Handler 的消息未处理完或者消息本身是延时消息的话，就会导致 Activity 退出后，从 Activity 到 Handler 到 Message 到 Looper 的引用链条一直存在，从而导致 Activity 的泄露！

  2. 单利模式引起的

    如果单利模式的构造需要传递一个 Context 对象，但是我们却传给它一个 Activity，这样就会导致这个单利对象会一直引用这个 Activity 对象。

  3. Thread 引起的

    比如线程是 Activity 的内部类，则线程对象中保存了 Activity 的一个引用，当线程的 run 函数耗时较长没有结束时，线程对象是不会被销毁的，因此它所引用的老的 Activity 也不会被销毁，因此就出现了内存泄露的问题。

  4. Android 组件引起的

    BraodcastReceiver，ContentObserver，FileObserver，Cursor，Callback等在 Activity onDestroy 或者某类生命周期结束之后一定要 unregister 或者 close 掉，否则这个 Activity 类会被 system 强引用，不会被内存回收。

  5. 不良代码

    bitmap 没有 recycle 等。

优秀实践：

  1. 对activity等组件的引用应该控制在activity的生命周期之内； 如果不能就考虑使用getApplicationContext或者getApplication，以避免activity被外部长生命周期的对象引用而泄露

  2. 在代码复审的时候关注长生命周期对象：全局性的集合、单例模式的使用、类的static变量等等。

  3. 尽量不要在静态变量或者静态内部类中使用非静态外部成员变量（包括context），即使要使用，也要考虑适时把外部成员变量置空；也可以在内部类中使用弱引用来引用外部类的变量；

  4. Handler的持有的引用对象最好使用弱引用，资源释放时也可以清空Handler里面的消息。比如在Activity onStop或者onDestroy的时候，取消掉该Handler对象的Message和Runnable：

  5. 线程Runnable执行耗时操作，注意在页面返回时及时取消或者把Runnable写成静态类。
    a) 如果线程类是内部类，改为静态内部类。
    b) 线程内如果需要引用外部类对象如context，需要使用弱引用。

  6. 在Java的实现过程中，也要考虑其对象释放，最好的方法是在不使用某对象时，显式地将此对象赋空，如清空对图片等资源有直接引用或者间接引用的数组（使用array.clear();array = null），最好遵循谁创建谁释放的原则。

避免Overdraw

  1. 写合理的布局。比如有一个 ListView，要给 item 下面添加一个下划线装饰。有的程序员为了图方便可能会采用在外面套一个布局，再设置布局背景的方式来达到效果，但是这既增加了层次，又产生了 Overdraw。有聪明一点的，可能会采用 RelativeLayout 来实现，不增加布局层次，也不产生 Overdraw，但是布局的测量次数会变多。其实可以使用 RecyclerView，它提供了添加 item 装饰器的功能。
  2. 去除不必要的背景。想想 View 的绘制流程，首先绘制的就是 background，从 ViewRoot 到各个子 View，基本上每个 View 都要绘制一遍自己的背景。通常我们的应用界面都会要求设置一个符合主题的背景，所以窗口（DecorView）的背景就没有必要，可以去除。还有就是各个界面控件之间的背景，肯定会有覆盖的，能够去掉的就去掉，这些都是细致活。
  3. 自定义控件的时候，有些比较复杂的效果，使用 clipRect 用来减少绘制区域，比如 DrawLayout 就做了这样的优化，当抽屉打开的时候，抽屉遮住的主界面部分是不会绘制的。

http 协议的无状态和无连接。

无连接的含义是限制每次连接只处理一个请求。服务器处理完客户的请求，并收到客户的应答后，即断开连接。采用这种方式可以节省传输时间。随着时间的推移，网页变得越来越复杂，里面可能嵌入了很多图片，这时候每次访问图片都需要建立一次 TCP 连接就显得很低效。后来，Keep-Alive 被提出用来解决这效率低的问题。Keep-Alive 功能使客户端到服务器端的连接持续有效，当出现对服务器的后继请求时，Keep-Alive 功能避免了建立或者重新建立连接。

无状态是指协议对于事务处理没有记忆能力，服务器不知道客户端是什么状态。即我们给服务器发送 HTTP 请求之后，服务器根据请求，会给我们发送数据过来，但是，发送完，不会记录任何信息。Cookie可以保持登录信息到用户下次与服务器的会话，换句话说，下次访问同一网站时，用户会发现不必输入用户名和密码就已经登录了（当然，不排除用户手工删除Cookie）。而还有一些Cookie在用户退出会话的时候就被删除了，这样可以有效保护个人隐私。与 Cookie 相对的一个解决方案是 Session，它是通过服务器来保持状态的。
