面试-四大组件

Activity生命周期？

一般都是面试的第一个问题，很简单的。但是也有衍生出一些奇葩问题，比如：

onStart() 和 onResume() 方法的区别？

onStart() 界面处于用户可见状态，比如有个 Dialog 样式的 Activity 浮在当前 Activity 上面，这个 Activity 就处于可见，但是不可交互状态。走到 onResume() 界面处于用户可交互状态。

什么情况下 onCreate() 方法执行了，onResume() 方法却没有执行？

这种简直就是脑筋急转弯，生命周期的方法中 crash 了后面的肯定不执行。

Fragment 生命周期

http://blog.csdn.net/forever_crying/article/details/8238863/

onAttach 和 onDetach 在最前和最后。不要忘记 onDestroyView。

Service生命周期？

启动 Service 的方法有两种，一种是 startService()， 另一种是 bindService()。这两种启动方式有不同的生命周期。

衍生问题：

多次调用 startService() 或者 bindService() 方法，生命周期是什么样子的？

调用多次 startService() 就会执行多次 onStartCommand() 方法，而调用多次 bindService()，只会执行一次 onBind() 方法。

startService() 启动的 Service 和 bindService() 启动的 Service 有什么区别？

生命周期不同。startService() 启动的 Service 是会一直运行下去的，除非客户端调用了 stopService 或者 Service 自己终结了自己。bindService() 启动的 Service 是和客户端绑定的，在客户端的 context 不存在之后，Service 会自动调用 onUnbind() 和 onDestory() 方法终结自己。

什么情况下既要 startService() 又要 bindService()？

如果我们需要开启一个长期在后台运行的 Service，又要和 Service 通信的话，就需要同时执行这两个方法。因为 bindService() 的 onBind() 方法会返回一个 IBinder 对象，可以用来和 Service 通信。注意结束 Service 的时候，需要同时调用 stopService() 和 unBindService()，并没有先后顺序。

Activity的几种LaunchMode及使用场景？

standard 模式
这是默认模式，每次激活Activity时都会创建Activity实例，并放入任务栈中。使用场景：大多数Activity。

singleTop 模式
如果在任务的栈顶正好存在该Activity的实例，就重用该实例( 会调用实例的 onNewIntent() )，否则就会创建新的实例并放入栈顶，即使栈中已经存在该Activity的实例，只要不在栈顶，都会创建新的实例。使用场景如新闻类或者阅读类App的内容页面。

singleTask 模式
如果在栈中已经有该Activity的实例，就重用该实例(会调用实例的 onNewIntent() )。重用时，会让该实例回到栈顶，因此在它上面的实例将会被移出栈。如果栈中不存在该实例，将会创建新的实例放入栈中。使用场景如浏览器的主界面。不管从多少个应用启动浏览器，只会启动主界面一次，其余情况都会走onNewIntent，并且会清空主界面上面的其他页面。

singleInstance 模式
在一个新栈中创建该Activity的实例，并让多个应用共享该栈中的该Activity实例。一旦该模式的Activity实例已经存在于某个栈中，任何应用再激活该Activity时都会重用该栈中的实例( 会调用实例的 onNewIntent() )。其效果相当于多个应用共享一个应用，不管谁激活该 Activity 都会进入同一个应用中。使用场景如闹铃提醒，将闹铃提醒与闹铃设置分离。singleInstance不要用于中间页面，如果用于中间页面，跳转会有问题，比如：A -> B (singleInstance) -> C，退出时是：C -> A -> B。

Android中跨进程通讯有几种方式？

4种，刚好分别对应着 Android 的四大组件。

Activity 之间的跨进程通信

比如，点击按钮拨打电话的功能。首先需要另一个进程的 Activity 支持跨进程启动，再加上另一个进程 Activity 设置的 Action 就可以实现跨进程通信。

Intent 的 action 是与清单文件中的对应，而 data 与 extra 都是传递的附加信息。

ContentProvider 的跨进程通信

我们获取短信，联系人等信息，就是通过 ContentProvider 方式实现的。一般 ContentProvider 是用来跨进程通信的，但是也可以用来进程内通信。可以将 ContentProvider 理解为数据访问层，要实现自己的 ContentProvider 必须要继承 ContentProvider，所以必须要实现其中访问数据的方法，这些方法对外暴露，看起来像是对数据库的操作，但是其实只是隐藏了底层的细节，因为你可以私底下在那些方法中对一个文件来进行操作（利用MatrixCursor）。一般我们的应用储存数据都是使用数据库，而为了向业务层提供数据，我们自己也会写一个访问数据的层，ContentProvider 的作用和这个类似，只不过它还支持跨进程通信而已。另外，在进行大数据量的跨进程传输的时候，ContentProvider 是最好的选择，因为其它形式的跨进程通信都是基于 IPC，而这些形式的跨进程通信效率都不高，并且还可能有数据量限制。

Broadcast 的跨进程通信

比如监听短信的到来。实现广播的方式都两种，但是都需要继承 BroadcastReceiver。注册方式分为两种，一种是静态注册，一种是动态注册，动态注册的一定需要解注册。安装应用的时候，广播接收者都是向 AMS 注册了的，广播发送的时候，也是向 AMS 发送，这样就形成了生产者消费者模式。

Service 的跨进程通信

ADIL的使用。使用 bindService() 方法，即可以绑定一个远程服务，与其进行通信。AIDL是通过 Binder 来实现跨进程通信的。

AIDL理解

当我们想要绑定一个 Service 的时候，该 Service 必须是要在 ServiceManager 里面注册过了的，我们调用了 bindService() 之后，ServiceManager 就会寻找目标 Service，然后向 Client 端返回一个 proxy 对象，这个 proxy 对象用来代理一切 Client 和 Service 的通信工作，同事也简化了 Client 端的调用操作（比如对数据封装成一定的格式满足跨进程通信的协议），这里是一个代理模式。还有一个需要注意的就是，如果是在同一个进程中使用 AIDL 那么，ServiceManager 返回的就不是 proxy 代理对象，而是一个本地对象，这里是策略模式。上面是文字描述过程，可能有一定了解的人才能看得懂，下面解释代码：
