饿汉式

最简单的单利模式，并且没有多线程问题。

因为在 Singleton 这个类被加载的的时候，类中的静态变量早就已经被初始化完毕了，所以调用 getInstance 方法的时候，返回的是一个完好的 instance 对象，唯一的缺点就是，如果 Singleton 实例化消耗较大，但是却又不能保证会使用到，这样就浪费了资源。

  1. 给 instance 分配内存
  2. 调用 Singleton 的构造函数来初始化成员变量
  3. 将instance对象指向分配的内存空间（执行完这步 instance 就为非 null 了）

但是在 JVM 的即时编译器中存在指令重排序的优化。也就是说上面的第二步和第三步的顺序是不能保证的，最终的执行顺序可能是 1-2-3 也可能是 1-3-2。如果是后者，则在 3 执行完毕、2 未执行之前，被线程二抢占了，这时 instance 已经是非 null 了（但却没有初始化），所以线程二会直接返回 instance，获取的是一个未初始化完成的对象。

改进方式三

volatile 除了可以保证变量的可见性之外，还可以禁止指令重排序优化。

也就是说，在 volatile 变量的赋值操作后面会有一个内存屏障（生成的汇编代码上），读操作不会被重排序到内存屏障之前。比如上面的例子，取操作必须在执行完 1-2-3 之后或者 1-3-2 之后，不存在执行到 1-3 然后取到值的情况。从「先行发生原则」的角度理解的话，就是对于一个 volatile 变量的写操作都先行发生于后面对这个变量的读操作（这里的“后面”是时间上的先后顺序）。

静态内部类

静态内部类，可以保证，如果没有用到该类，该类不会被加载，因此静态变量也不会被初始化。

如果用到了，在加载类的时候，静态变量就已经被初始化好了，不会有多线程问题。
