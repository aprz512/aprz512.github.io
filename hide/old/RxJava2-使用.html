RxJava 是一种增强的观察者模式，具有非常强的通用性。

下面我就基于观察者模式来说一下我对RxJava的理解。

如何使用

  1. 创建被观察者

  2. 创建观察者

    也有别的方法创建观察者，可以点开 Observerable 的 subscribe 方法看看它接受哪些类作为参数。

    这里说一个题外话，比如我们使用 Consumer 作为观察者的话，需要注意一些地方：

    如果你使用了一个 Consumer 作为参数，那么这个 Consumer 只能处理 onNext 方法，假如你的被观察者发送了 onError，那么就会报错（OnErrorNotImplementedException）。

  3. 订阅

    subscribe() 这个方法有点怪：它看起来是『observalbe 订阅了 observer 』而不是『observer订阅了 observalbe』，这看起来就像『杂志订阅了读者』一样颠倒了对象关系。这让人读起来有点别扭，不过如果把 API 设计成 observer.subscribe(observable) ，虽然更加符合思维逻辑，但对流式 API 的设计就造成影响了，比较起来明显是得不偿失的。

  4. 解除订阅

    • observable 的 subscribe 会返回一个 disposable 对象，调用这个 disposable 的 dispose 方法就可以解除订阅关系了。
    • 如果你手动调用了 onComplete 方法，那么等到观察者的方法被回调的时候，会自动解除订阅关系。

    • 有多个observer怎么办

    解除订阅关系之后，被观察者仍然可以发送数据，只不过观察者收不到了。

    subscribeOn 可以先简单理解为数据产生所在的线程。

    observeOn 可以先简单理解为观察者回调方法执行的时候所在的线程。

  • 复杂的线程切换

    这里先说结论：

    • 多次指定发射事件的线程只有第一次指定的有效，也就是说多次调用 subscribeOn() 只有第一次的有效，其余的会被忽略。

调度器介绍

  1. Scheduler.io

    用于IO密集型任务，如异步阻塞IO操作，这个调度器的线程池会根据需要增长，没有边界；对于普通的计算任务，请使用Schedulers.computation()；

  2. Schedulers.computation( )

    用于计算任务，如事件循环或和回调处理，不要用于IO操作(IO操作请使用Schedulers.io())；默认线程数等于处理器的数量，避免cpu饥饿。

  3. Schedulers.newThread( )

    为每个任务创建一个新线程，使用的时候要注意，避免耗尽资源。

  4. Schedulers.immediate( )

    在当前线程立即执行。

  5. Schedulers.single()

    永远都只会在同一个子线程中执行。

  6. AndroidSchedulers.mainThread()

    它指定的操作将在 Android 主线程运行。

注意：io 与 computation 都是有自己的线程池的。computation 的线程池比较有意思，它是每一个 worker 对应着一个线程池（一共有cpu核心数个worker），但是这个线程池里只有一个核心线程，由于等待队列是无界的，所以相当于永远只有一个线程会运行。

io 和 computation 的线程池都不是直接使用的系统的线程池，而是自己管理的。

背压的处理

有时候我们会遇到这样的情况：

被观察者 发送事件速度太快，而观察者 来不及接收所有事件，从而导致观察者无法及时响应 / 处理所有发送过来事件的问题，最终导致缓存区溢出、事件丢失 & OOM 。

解决方法是使用 Flowable，它提供了背压策略。
