最近在公司的项目中参与了框架搭建的工作，说一些自己的心得与感受。

首先从图片框架搭建说起，这里并不是说去自己写一个图片框架，而是说抽取出一个通用的图片接口层，起初我自己只是有一个大致的方向，就是要用策略模式来做出一个可以快速替换图片加载框架实现的一个接口层。但是刚开始起步的时候，我想了很久，觉得没法开始，所以我就去github上去找看看别人的思路，但是看了很多，都达不到我的心里预期，最后一狠心决定还是自己边敲边改吧。

这里不得不说一下，上面的《滴滴国际化项目 Android 端演进》给我的启发，在我自己码完自己的框架的时候，还是有点小得意的，但是我突然想到了滴滴地图sdk适配的场景，这特么不是和我的需求一样么，于是我就又抽取了一层接口，做了一个适配器。在没有想起这个场景之前，我是觉得哪里有点不足，但是找不到问题。

第二个就是 dataBinding 的引入。这个纯粹是一个巧合吧，本来公司的项目是 MVP 模式的，但是说是想尝试一下新的东西，所以就引入了，但是使用的不深，暂时只是当做 ButterKnife 来用，我就想，既然要尝试新的东西，那就不妨使用 MVVM 模式吧。

由于我从来没有接触过这方面的东西，所以只能自己去查相关资料，找demo。在经过一段时间的查阅之后，我勉强仿写出了一个 bilibili 登录界面的 demo。首先，我说说自己的使用感受，在某种意义上来说，MVP 和 MVVM 只是名字不一样，但是 MVVM却不用写接口去与 View 层交互。

其实我之前使用 MVP 最大的感受就是：

  1. 只是将 Activity 或者 fragment 中的逻辑代码搬到了 presenter 中，或许是我们写的不太规范，但是确实有很多人这样用。
  2. presenter 层涉及到多个网络请求，或者数据库操作的时候，有的人还会加入一些Model的逻辑代码进去，就是说由presenter去控制该去哪里取数据。
  3. 当 View 变化的时候，接口也要跟着变化，但是用到接口的所有地方都要变化，引起连锁反应，这时我就感觉，这样的解耦有什么用？？
  4. presenter 的生命周期超越了 Activity等的生命周期，导致每次回调的时候都需要判断。

其实在我的理解中，presenter 与 controller 应该一样，都是不涉及任何逻辑代码的，他们只负责转发。所以在我看到 google Android developer doc 架构设计教程中的思路时，简直太感动了：https://developer.android.com/jetpack/docs/guide#common_problems_faced_by_app_developers。

他们是这样设计的，presenter 层不是直接的去与 Model 数据交互，而是通过一个接口层：Repository，由它来实现一个View所有的数据访问接口。它的里面又会有不同的委托实现，比如从网络请求，我们叫 Remote 实现，比如从数据库请求，我们叫 Local 实现，至于什么时候去访问网络，或者访问那个网络，都是放在这个 Repository 中的，与外界隔绝，这样就保证了类职责的单一，到时候页面变化了，我们可能就只需要替换 remote 或者 local 实现，或者只是变一下访问逻辑。只要数据类型不变化，提供给presenter的接口就不用变化。

最后我还想说一下，这个是我想起了我去脸萌面试的一个经历，当时我去面试的时候，他们是想招一个架构师，但是我对这个涉猎不深，问的问题大都答不出来，但是我还是厚着脸皮问了答案，那个面试官挺不错，给我说了他自己的看法。其中我最有印象的是这样的一个概念，就是说无论是 MVC 还是 MVP，不能局限与书本，书上写了 MVC 中 M 与 C 直接交互，但是我虽然类的名字叫 MVC 但是我不让 M 与 C 直接交互，那它是什么模式呢？无论类的名字是什么，只要你的设计是好的，能够提供更好的结构，搭建出好的框架就行。其实我的意思是说，在我们使用一个模式的时候，看到别的模式有好的地方，都可以引进来，不要纠结于某个名称。
