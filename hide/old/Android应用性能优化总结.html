正好有个构建自己知识图谱的一个打算，所有我就将书上的东西整理一下，便于之后回忆。

如果你想知道一个变量是怎么传递过来的，这个就非常有用了，特别是嵌套特别深的时候。

第二章

这一章就是说的绘制优化，主要是解决卡顿和监控卡顿。

Android显示原理可以简单概括为：Android应用程序把经过测量、布局、绘制后的 surface 缓存数据，通过 SurfaceFlinger 把数据渲染到显示屏幕上，通过Android的刷新机制来刷新数据。应用层负责绘制，系统层负责渲染。

Android 的刷新机制：

vsync 信号，4.1之前应用层也要注册 vsync 信号。

triple buffer，大部分情况下都是双缓冲，只有在需要的时候才用到。

有需要的可以去查一下时序图，16ms。

性能分析工具：

Profile GPU Rendering

TraceView

Systrace UI

布局优化工具：

Hierarchy Viewer：可以查看层级和耗时

Android Lint：可以全局分析布局，特别是 TooDeepLayout，TooManyViews

布局优化方法：

减少层级，层级相同的情况下使用 FrameLayout 或者 LinearLayout，减少层级使用 RelativeLayout。

使用 merge，特别是封装通用的 viewGroup，如 titleBar 等。

提高界面的显示速度，可以使用 viewStub，xml中有些控件在某些情况下不需要显示或者延迟显示可以使用这个。

布局复用，这个算是代码层面的优化。

避免过度绘制：

检查：开发者选项 -> 显示GPU过度重绘

布局上的优化：如果 fragment 有自己的背景，可以考虑去除 Activity 的背景（默认就有一个）。

自定义view的优化：使用 clipRect 来制定绘制区域，避免绘制不需要绘制的地方，比如重合的地方。

启动优化：

监测启动时间

启动界面布局优化，同上。

启动逻辑进行分类：必要、非必要、耗时、不耗时。有几种处理方式：分步加载，异步加载，延迟加载。

合理的刷新机制：

比如控件数据没有变化，或者控件不可见就没有必要刷新。

刷新也可以指定刷新的区域，invalidate(Rect rect)

避免后台线程的影响：比如 list 滑动的时候，暂停图片的加载等。（不知道为啥要放在这里）

提升动画性能：

补间动画会导致频繁的重绘，相比之下，属性动画性能更好。

硬件加速与离屏缓冲（LAYER_TYPE_HARDWARE），离屏缓冲会在硬件加速的基础上再次提高性能，不过只适用于平移，旋转，缩放，透明度等，你可以认为它是保存了一个 Bitmap，如果你的动画效果导致缓存失效（引起了重绘），从而浪费了缓存，还降低了性能（因为每次都要重新绘制缓存Bitmap）。

卡顿监控方案：

http://www.10tiao.com/html/330/201801/2653579565/1.html

有很多，我就不哔哔了。在卡顿时，我们可以获取：整体 CPU 的使用率和当前应用进程的占用率。

第三章

这一章主要是讲了内存优化。

Android内存的管理机制：

java对象的生命周期：7个阶段

内存分配，内存回收机制，主要是 GC。

内存优化的意义：

内存泄露是一方面，即使内存不泄露，在短时间内创建大量的对象，也会引发 GC 的运行，而 GC 的运行会挂起其他线程（即使 art 优化了，也会有短时间的挂起），如果 GC 频繁的运行，就会导致界面卡顿。

所以，有几个方面的好处：减少OOM，减少卡顿，减少内存占用，提高后台运行时的存活率，减少异常发生（代码隐患）。

内存分析工具：

Memory Monitor：主要用来监测大内存申请和内存抖动，很直观

Heap Viewer：查看各个数据类型占用内存的情况。

Allocation Tracker：可以记录应用程序的内存分配，并且能够查看调用堆栈。

MAT：分析内存泄露。可以查看对象到GC ROOTS 的最短路径，可以比较内存映像。

常见内存泄露场景：

资源未关闭

注册对象未注销

类的静态变量持有大数据对象

非静态内部类的静态实例

Hander 临时性内存泄露（原因同上）

容器中的对象没有清理（非静态的 ThreadLocal 使用）

webview

内存泄露监控：

LeakCanary，通常要了解一下原理。

优化内存空间：

对象的引用（四种引用方式，虚引用可以用来做析构函数一样的东西）

减少不必要的开销：自动装箱（特别要注意循环中），内存复用（listview，Bitmap，对象池）

选择数据类型：

HashMap 与 ArrayMap：少于1000的，插入删除不频繁的，可以使用 ArrayMap，节省内存。

尽量不要使用枚举，当类型较多不好控制的时候，可以使用注解：@IntDef 和 @StringDef

使用 LruCache

图片内存优化：

设置解码格式 inPreferredConfig（8888,4444等），注意一下，使用 decodeResource 来解码图片的时候，设置之后可能不起作用。

设置 inSampleSize 如果缩放倍数不是 2 的整数幂，可以使用 inScaled，inDensity，inTargetDensity

使用 inBitmap，注意 4.4 之前和之后的区别。（bitmap 的复用其实就是bitmap 占用了两块空间，一个是 java 层（对象），一个是 jni 层（像素），复用的是 jni 层的空间）。

使用开源图片加载项目，如果你能读懂源码就更好了（本质上不会变，可以学习设计模式）。

第四章

这一章讲的是储存优化。

储存方式：

SharedPreference，文件，SQLite，ContentProvider

序列化：

Serializable：在序列化的时候会产生大量的临时变量

Parcelable：无法储存在磁盘上

Gson

Nano Proto Buffers

FlatBuffers：性能最好

SP 优化：尽量使用 apply 代替 commit，批量操作可以先获取 editor，执行批量操作，再 apply。

数据库优化：

SQLiteStatement

使用事务，我们的每一个操作默认会开启一个事务，如果不显示的使用，会导致频繁的开始事务。

建立索引（增加空间占用，删除插入操作变慢）

异步线程，写数据库的同一管理。就是将操作封装起来，放到一个队列里等待执行

提高查询性能，不要查询不需要的字段。

网络连接：使进程保持活动状态
