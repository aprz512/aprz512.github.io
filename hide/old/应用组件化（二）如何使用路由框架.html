应用为什么要组件化？

  1. 实际业务变化非常快，但是工程之前的业务模块耦合度太高，牵一发而动全身.
  2. 对工程所做的任何修改都必须要编译整个工程，编译一次花3~5分钟是很正常的
  3. 功能测试和系统测试每次都要进行.
  4. 团队协同开发存在较多的冲突.不得不花费更多的时间去沟通和协调,并且在开发过程中,任何一位成员没办法专注于自己的功能点,影响开发效率.
  5. 不能灵活的对工程进行配置和组装.比如今天产品经理说加上这个功能,明天又说去掉,后天在加上.

上面都是我从别人的博客 copy 过来的，但是我也是深有体会。

第一种情况是：自己改某个bug的时候，一旦某个 bug 还需要不断的调试，就陷入一种循环，打一个包，看一下效果，改代码，再打包，看效果….不知不觉的一个小时就没了，全部时间都花在了打包上。回过头来想想其实bug很简单，本来是没有必要花这么长时间的。

第二种情况是：产品经理要加上某一功能，但是到了上线的时候又说暂时不上这个功能，所有得给这个模块功能加上一个开关，但是由于原来的项目耦合非常的严重，导致要加开关的地方非常多（入口多），就会导致代码出现很多判断，显得很乱。

组件如何实现单独调试和发布？

从上面的架构图来看，我们应用的每一个模块（大的功能）都是一个组件，这个组件与功能的不同之处在于，它既可以单独运行又可以当做一个依赖库。为什么要这样做呢？第一是模块可以被重用，比如，你的某个模块想要接入到别的应用里面，就可以直接打包一个 aar 给别人使用。第二个是便于修改和测试，原来我们的项目结构是只有一个工程，只要对项目的某一个地方修改了，整个项目就需要全部重新编译一次（Instant Run 在某些情况下不好使），会花掉比较长的时间，组件化了之后，修改了这个组件的东西，就只需要运行这个组件就好了，测试和改bug都很快。

组件之间的通信

虽然我们的项目做了组件化，但是组件之间的交互还是有的，比如，从A组件跳转到B组件，这种很常见。

这个时候就需要用到路由框架了。

由于组件之间是禁止相互依赖的，所以如果你想要实现跳转功能，只能借助于路由框架。可以使用ARouter，虽然JIMU也有自己的路由框架，但是暂时还没有支持fragment。

在使用 ARouter 的过程中，我们遇到的问题是：如果两个模块之间需要传递参数该怎么处理？

因为如果只是传递简单的数据类型都好说，如果我们有一个 Song 变量，它代表的一首歌，那么我从A模块跳转到B模块的时候，需要传递这个 Song 变量的时候，该怎么做？注意这里 A 和 B 是没有依赖的，也就是 A 模块有个 javaBean 叫 Song，但是 B 模块没有这个 javaBean，就算有它们也不是同一个类。

我们的解决方案是将这个公用的类抽取出来，放到一个单独的module中。后来我看了JIMU中的解决方法，发现它也是这样做的，并且它还约束了模块之间的通信接口。

资源与依赖管理

之前我们做这一块的时候有过讨论，就是说如果每个组件的资源都是自己管理的话，那么肯定会有一些重复的东西，但是说将资源抽离出来放到一个单独的 module 里面，这个里面只有资源啥都没有，这样确实能解决一些问题，但是却失去了组件的可重用性。所以我们决定还是每个组件维护自己的资源。

再说依赖的管理，我们现有的工程是将每个基础库都抽成了一个lib（主要是为了单独复用，我觉得有点过了），比如图片加载是一个lib，日志是一个lib，权限是一个lib，网络是一个lib，但是这样做有很多问题，就是这些lib还需要去依赖一些更基础的东西，比如 util 等，但是这样做肯定是不好的，我们的解决方法就是lib里面维护自己的util，即使有重复的，所以我个人认为将这个东西放在一个 module 里面会好些，这个 module 就相当于一个 sdk，可以用来进行快速开发，虽然可能有些东西用不到，但是这个sdk只放一些必要的就好了。
