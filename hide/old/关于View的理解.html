View的绘制流程

View 的核心绘制流程从 ViewRootImpl 的 performTraversals() 这个方法开始。该方法里面的 measure()， layout()， draw() 方法实现了整个 View 树的绘制。

首先看 View 的measure() 方法，该方法是 final 的，子类无法覆盖，但是它提供了一个 onMeasure() 方法用来让子类测量自己的宽和高。onMeasure() 方法有两个参数，一个 widthMeasureSpec，另一个 heightMeasureSpec，是父布局的传递过来的，父布局自己的参数。这两个都是 int 类型的值，一共占 32 位，前 2 位表示测量模式，后 30 位表示测量大小。一般来说，子类测量自己的宽和高的时候，都需要考量父布局的参数，不能超过父布局限定的宽和高等等。最后就是，子类在覆盖 onMeasure() 方法的时候，必须要调用 setMeasuredDimension() 方法，将测量的宽和高传递进去。此时，View 的测量宽和高才会被赋值。

再看 View 的 layout() 方法，layout() 有四个参数，分别表示距离父布局 l, t, r, b 的距离。该方法首先会判断，传递进来的四个参数，与原先的四个参数是否一样，如果不一样就表示控件的位置移动了，就会调用 onLayout() 方法。当我们自定义 ViewGroup 的时候，就需要在 onLayout() 方法中来放置 child view 的位置。比如 LinearLayout ，当排列方向为竖直方向的时候，我们就要根据 padding，margin，gravity 计算每一个 child 的 l 和 t，然后加上测量的宽高算出 r 和 b。这样 View 的布局就完成了，向 getTop() 等方法才会有值。

最后是 View 的 draw() 方法，首先是对 View 的背景进行绘制，然后对 View 的内容进行绘制，也就是调用 onDraw() 方法，该方法需要子类自己来实现。再接着就是绘制该 View 的 Child，最后，绘制滚动条。似乎并没有什么好说的。

关于 View 的重绘，当我们调用了 invalidate() 方法之后，实际上就产生了一块 dirty 区域，就是调用 invalidate() 方法的 View 的矩形区域，而 invalidate() 内部会不断的往父布局回溯，每回溯一层就重新计算一下，将当前层的区域与 children 的 dirty 区域求并集，最终会得到一个 dirty 区域。Window 的显示其实是使用了 Surface ，Surface 类是使用一种称为双缓冲的技术来渲染应用程序窗口的UI的。这种双缓冲技术需要两个图形缓冲区，其中一个称为前端缓冲区，另外一个称为后端缓冲区。前端缓冲区是正在渲染的图形缓冲区，而后端缓冲区是接下来要渲染的图形缓冲区。在渲染后端缓冲区的时候，会将后端缓冲区与 dirty 区域做一个交集，得到的结果就是需要重新渲染的区域，其他区域就直接从前段缓冲区那里拷贝过来，不用重新渲染。如果不能将前端缓冲区的内容拷贝到后端缓冲区中去，那么接下来就需要重新绘制应用程序窗口的所有区域了。这就是重绘的大致流程。

另外，关于 Choreographer 这个类，它主要是用来做这样的工作的，Choreographer 里面管理一个消息队列，每当有界面请求重绘的时候，实际上是向 Choreographer post 了一个 Runnable 对象，这个 Runnable 里面包含着 performTraversals() 方法。当 Choreographer 收到了这个对象之后，并不是立即的去执行，而是去等待 VSYNC 信号的产生，等到 VSYNC 信号产生的时候，再去执行 Runnable 里面的方法。Choreographer 里面有个 doFrame() 方法，用来同步 VSYNC 信号和每一帧渲染时间，处理丢帧等情况。

Android中的几种动画

第一种是比较容易的帧动画，一般用作 loading 动画的显示，原理是将在 xml 设置的 drawable 全部加载进来，然后一帧只显示其中一个 drawable，如此循环，就像电影一样。

第二种是稍微复杂一点的渐变动画，用作对 View 进行平移，旋转，缩放，透明度变化等等。可以在 xml 中实现，也可以用 java 代码实现。xml 中为每个动画都提供了属性值，用来设置动画，比如缩放，你需要设置，从那个点开始缩放，x，y 缩放多少（值的表示有3种形式，数值、百分数、百分数p，譬如50表示以当前View左上角坐标加50px为初始点、50%表示以当前View的左上角加上当前View宽高的50%做为初始点、50%p表示以当前View的左上角加上父控件宽高的50%做为初始点），缩放时间等等。需要注意的就是，渐变动画并不会改变 View 的透明度，位置等属性，如果一个控件采用渐变动画移动了位置，尽管看起来控制并不处在原来的位置，但是其点击事件响应区域仍然在原来的位置，记住这只是幻觉。

第3中就是可以改变 View 属性的动画，属性动画。比如 ObjectAnimator，如果你想改变 View 的 XXX 属性，并不是简单的渐变动画，就可以使用这个，同时，需要该 View 实现 setXXX() 方法，这样等该动画完毕之后，其属性也完全变了。同样拿平移举例，使用 ObjectAnimator 会更改 X , Y 值，点击事件的相应区域会跟随 View。我最常用的是 ValueAnimator，之前就做过这样一个控件，一个圆形按钮，旁边有一圈进度条，可以根据人说话的声音来调整进度条的位置。

最后，说一下插值器，其实插值器的原理比较简单。假设我们设置好了一个动画效果，其时间为 T，动画内容是翻转 S 角度，这里我们给动画设置了一个 加速减速插值器，那么它是怎么工作的呢？

解释这行代码之前，我们考虑这样的一个问题：如果主线程的消息非常多的话，即使我们的应用 ui 层次很简洁，CPU 可以在 16ms 之内完成，但是怎么保证别的消息不会阻碍绘制消息的运行？？？

这个同步屏障的作用可以理解成拦截同步消息的执行，主线程的 Looper 会一直循环调用 MessageQueue 的 next() 来取出队头的 Message 执行，当 Message 执行完后再去取下一个。当 next() 方法在取 Message 时发现队头是一个同步屏障的消息时，就会去遍历整个队列，只寻找设置了异步标志的消息，如果有找到异步消息，那么就取出这个异步消息来执行，否则就让 next() 方法陷入阻塞状态。如果 next() 方法陷入阻塞状态，那么主线程此时就是处于空闲状态的，也就是没在干任何事。所以，如果队头是一个同步屏障的消息的话，那么在它后面的所有同步消息就都被拦截住了，直到这个同步屏障消息被移除出队列，否则主线程就一直不会去处理同步屏幕后面的同步消息。

而所有消息默认都是同步消息，只有手动设置了异步标志，这个消息才会是异步消息。另外，同步屏障消息只能由内部来发送，这个接口并没有公开给我们使用。

最后，仔细看上面 Choreographer 里所有跟 message 有关的代码，你会发现，都手动设置了异步消息的标志，所以这些操作是不受到同步屏障影响的。这样做的原因可能就是为了尽可能保证上层 app 在接收到屏幕刷新信号时，可以在第一时间执行遍历绘制 View 树的工作。

因为主线程中如果有太多消息要执行，而这些消息又是根据时间戳进行排序，如果不加一个同步屏障的话，那么遍历绘制 View 树的工作就可能被迫延迟执行，因为它也需要排队，那么就有可能出现当一帧都快结束的时候才开始计算屏幕数据，那即使这次的计算少于 16.6ms，也同样会造成丢帧现象。
